## Purpose

This addendum aligns the APW implementation plan with the root website intent canon.

---

## Implementation sequencing correction

Future APW implementation work must not begin with visual cards, isolated page sections, or generic services copy.

Implementation must first preserve the website spine:

1. loss-prevention proposition;
2. malicious and non-malicious loss framing;
3. maturity assessment and gap-identification journey;
4. free maturity assessment CTA and ISMS/APGI Hub handoff;
5. ecosystem role mapping;
6. services, training, case studies, and contact paths in support of that journey.

---

## Platform implementation obligation

When implementing or correcting the Platform/APGI Hub page, the builder must implement a journey-led page structure before or alongside module-card rendering.

The page must make clear that MMM, Risk Management, PIT, Incident & Intelligence, RADAM/Remote Assurance, Skills Development, and ISMS/APGI Hub support one maturity progression.

---

## Free assessment implementation obligation

The implementation plan must include a public free maturity assessment entry point.

If the final ISMS/APGI Hub assessment URL is unavailable, the builder must implement a polished pending or contact-assisted handoff state rather than omitting the assessment path.

---

## Evidence expectation

Build evidence must show where the maturity journey, loss framing, free assessment path, and ecosystem role mapping appear in the implemented site.

---

## Boundary

The builder must not implement private assessment logic, data persistence, CRM automation, user accounts, or full platform runtime inside APW unless separately approved.
