## Purpose

This addendum adds functional requirements that bind APW to the root website intent canon.

---

## New functional requirements

### APW-FR-INTENT-001 - Loss-prevention proposition

APW must explain that APGI helps organisations prevent loss by improving maturity and reducing opportunity generators.

### APW-FR-INTENT-002 - Malicious and non-malicious loss framing

APW must distinguish malicious loss from non-malicious loss in public-facing language and show that APGI's approach addresses both.

### APW-FR-INTENT-003 - Maturity journey spine

APW must present the maturity journey from assessment through roadmap implementation and resilience. This journey must be visible across the homepage and Platform/APGI Hub content.

### APW-FR-INTENT-004 - Free maturity assessment entry point

APW must include a clear free maturity assessment CTA or entry point. The CTA must explain that the free assessment provides an initial maturity indication and routes the visitor toward the ISMS/APGI Hub destination for deeper assessment.

### APW-FR-INTENT-005 - Ecosystem role explanation

APW must explain the roles of MMM, Risk Management, PIT, Incident & Intelligence Hub, RADAM/Data Analytics/Remote Assurance, Skills Development/Training, and ISMS/APGI Hub in the maturity journey.

### APW-FR-INTENT-006 - Platform page is not a flat module directory

The Platform/APGI Hub page must include narrative context that explains how modules work together. A grid of app cards alone is insufficient.

### APW-FR-INTENT-007 - Real-time situational awareness concept

APW must introduce the concept that mature organisations increasingly understand work activity, risk, controls, evidence, exceptions, and escalation before loss occurs or while weak signals are still manageable.

### APW-FR-INTENT-008 - Hover / click / ask / train pattern

Where interactive education is used, APW should support layered explanation: one-line summary, deeper detail, Maturion guidance where available, and training referral.

### APW-FR-INTENT-009 - Case study confidentiality

Case studies must be anonymised by default and must not include client or project names unless explicit permission is recorded.

---

## Functional boundary

These requirements do not approve APW to implement backend assessment logic, user accounts, Supabase persistence, CRM workflows, LMS administration, or private platform runtime functions.
