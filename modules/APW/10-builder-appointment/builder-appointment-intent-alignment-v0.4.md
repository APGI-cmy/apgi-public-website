# APW Stage 11 Addendum - Builder Appointment Intent Alignment v0.4

**Stage:** 11 - Builder Appointment  
**Authority:** CS2 - Johan Ras  
**Root intent anchor:** `APW_WEBSITE_INTENT.md`  
**Status:** Corrective alignment addendum

---

## Purpose

This addendum amends the Stage 11 builder appointment conditions so the appointed APW builder is bound to the root website intent canon.

---

## Builder appointment condition

Any APW implementation builder appointed after this addendum must treat `APW_WEBSITE_INTENT.md` as a required source artifact.

The builder is appointed to build against the corrected prebuild chain plus this root intent anchor.

---

## Required builder acknowledgement

Before implementation, the builder must acknowledge that APW must:

- present APGI as a holistic loss-prevention and maturity-improvement partner;
- explain malicious and non-malicious loss;
- present the maturity journey as the conceptual spine;
- include a free maturity assessment entry point and ISMS/APGI Hub handoff;
- explain ecosystem modules as connected journey components;
- preserve training, coaching, monitoring, remote assurance, and implementation tracking as maturity supports;
- use case studies only in anonymised form unless permission exists;
- avoid unapproved backend/platform runtime scope.

---

## Stop rules

The builder must stop and escalate if:

- the available content would reduce APGI to a generic services brochure;
- the Platform/APGI Hub page would become only a module-card grid;
- the free maturity assessment destination is missing or unclear;
- a requested implementation would require backend assessment logic, Supabase persistence, CRM, auth, LMS administration, or private ISMS runtime inside APW without approval;
- case-study content includes client or project names without permission.

---

## Stage 12 handoff condition

Future Stage 12 build evidence must demonstrate how the implementation satisfies this addendum and `APW_WEBSITE_INTENT.md`.
