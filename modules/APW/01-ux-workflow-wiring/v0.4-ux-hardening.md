# APW v0.4 UX Workflow Hardening

## Status

Drafted for independent review.

## Purpose

This artifact hardens Stage 2 so that later FRS and build work can define a fully functional public experience without guessing.

## Required Card Families

The v0.4 website must support these card families:

| Card family | Required use |
|---|---|
| Offering cards | Security Philosophy, Maturity Roadmap, Consulting, Remote Assurance, Digital Tools, Case Studies, Training. |
| Maturity cards | Basic, Reactive, Compliant, Proactive, Resilient. |
| Tool cards | Maturion, MMM, PIT, Risk, RADAM, Incident and Intelligence, Skills / Training. |
| Case-study cards | Public-safe proof patterns and anonymized examples. |
| Training cards | Classroom/contact and e-learning options. |

## Hover Requirements

Hover text must be short, plain-language, and curiosity-building.

Hover must never contain critical information that is unavailable on mobile.

Mobile fallback must show the same meaning through visible summary text or tap-accessible detail.

## Click Detail Requirements

Clicking a card must provide deeper context through a page section, drawer, modal, or dedicated detail surface.

Each detail surface must include:

- what the concept is;
- why it matters;
- where it fits in the APGI maturity journey;
- what the visitor can do next;
- an Ask Maturion or training/contact path;
- public-safe wording only.

## Ask Maturion Requirements

Ask Maturion must be present as a guided public CTA.

Until a public Maturion endpoint is approved, the Ask action may resolve to:

- a coming-soon explanation;
- a contact CTA;
- a training recommendation;
- an approved public Maturion/MMM destination.

The UX must not imply that private knowledge access or live AI chat is available before it is approved.

## Train Routing Requirements

Where a topic has a training implication, the UX must route the visitor toward:

- the APGI Training page;
- the public Thinkific storefront;
- a classroom/group delivery contact path.

Examples:

- maturity learning routes to Maturity Roadmap and Training;
- risk learning routes to Risk-related training or Contact;
- human rights learning routes to VPSHR training;
- project implementation routes to PIT, Consulting, or Training;
- remote assurance routes to Contact or Digital Tools.

## Page Section Requirements

| Page | Required sections |
|---|---|
| Home | Hero, maturity-led proposition, offering cards, Maturion prompt, training/contact CTA. |
| Security Philosophy | Operating view, maturity implication, real-world application, next steps. |
| Operational Maturity Roadmap | Five maturity levels, domains/MPS explanation, evidence explanation, next-step paths. |
| Consulting and Implementation | Advisory role, implementation support, coaching, PIT linkage, contact CTA. |
| Remote Assurance | Review model, control visibility, improvement support, digital tool linkage, contact CTA. |
| Digital Tools and Maturion ISMS | Tool cards, maturity role, Ask Maturion prompt, training/contact paths. |
| Case Studies | Anonymized cards, challenge/approach/result pattern, public-safe boundary. |
| Training | Classroom catalogue, e-learning catalogue, group/contact path. |

## No-Dead-End Rule

Every major section must provide at least one useful next step.

Allowed next steps:

- read another offering page;
- ask or register interest in asking Maturion;
- contact APGI;
- explore training;
- open Thinkific;
- view a public-safe case study.

## Accessibility Requirements

The UX must support keyboard access, focus states, mobile tap behavior, modal close controls, meaningful button labels, and readable long-card layouts.

## Stage 2 Acceptance Criteria

Stage 2 v0.4 is complete only when it defines:

- required card families;
- hover behavior;
- click/detail behavior;
- Ask Maturion fallback behavior;
- Train routing behavior;
- required page sections;
- no-dead-end rules;
- accessibility expectations;
- implementation gate.

## Gate

This is Stage 2 only. It does not authorize implementation.
