# APW v0.4 App Description Hardening

## Status

Drafted for independent review.

## Purpose

This artifact hardens the v0.4 app description so the next stages do not have to infer APGI's intent.

APW v0.4 must explain APGI as a maturity-led advisory, training, assurance, and technology-enabled organisation.

The website must present one journey: understand operating maturity, identify gaps, improve controls, implement change, verify evidence, and build capability through training and Maturion-supported tools.

## Public Value Proposition

APGI helps organisations move from informal or reactive operating states toward resilient operating states.

The public site must make clear that maturity is not a document library. Maturity requires observable practice, clear ownership, working controls, evidence, monitoring, learning, and improvement.

## Maturity Story

The website must explain five maturity states in plain language:

1. Basic: controls are informal and person-dependent.
2. Reactive: action happens mainly after events, findings, or pressure.
3. Compliant: required controls and evidence exist at a minimum acceptable level.
4. Proactive: risks, trends, controls, evidence, and improvement actions are actively managed.
5. Resilient: controls are embedded, monitored, recoverable, and supported by stronger systems and routines.

## Required Concepts

The website must explain these concepts without overwhelming the visitor:

- maturity domains;
- Minimum Performance Standards;
- criteria;
- maturity descriptors;
- evidence requirements;
- gap-to-next-level guidance;
- implementation tracking;
- remote assurance;
- training and capability building;
- Maturion as an AI guide across the ecosystem.

## Page Intent

| Page | Visitor should understand |
|---|---|
| Security Philosophy | APGI sees security as a maturity journey tied to work, people, controls, evidence, and improvement. |
| Operational Maturity Roadmap | The roadmap defines where the organisation is, what must improve, and what evidence is needed. |
| Consulting and Implementation | APGI helps translate maturity requirements into practical improvement work. |
| Remote Assurance | Independent review and data-informed assurance help keep controls visible and improving. |
| Digital Tools and Maturion ISMS | The apps support maturity, implementation, risk, assurance, workflow, and learning. |
| Case Studies | APGI's approach can be shown through public-safe patterns and anonymized examples. |
| Training | Training builds the capability required to operate and sustain maturity improvements. |

## Boundaries

APW v0.4 must remain a public website.

It may use approved summaries, public links, and governed placeholders.

It must not expose private source records, private client detail, private knowledge files, private app surfaces, or live AI integration before approval.

## Functional Build Intent

A future build is functional only if a visitor can:

- understand APGI's maturity-led model;
- move from the homepage into each offering area;
- use Hover -> Click -> Ask -> Train interactions;
- reach contact or training paths from every major page;
- understand Maturion's role without accessing private systems;
- see anonymized proof patterns without confidential disclosure;
- browse comfortably on desktop and mobile.

## Stage 1 Acceptance Criteria

Stage 1 v0.4 is complete only when it defines:

- the core APGI maturity-led proposition;
- the maturity story;
- the required public page areas;
- Maturion's public role;
- the knowledge boundary;
- the case-study boundary;
- functional build intent;
- downstream gates before implementation.

## Gate

This is Stage 1 only. It does not authorize implementation.
